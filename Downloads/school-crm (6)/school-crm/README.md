# מערכת ניהול יועץ חינוכי

מערכת פנימית לניהול תלמידים ופניות עבור יועץ/ת חינוכי בבית ספר.

## מה יש במערכת

- **מיפוי תלמידים (פורטל מורים + לוח בקרה ליועץ)** - מנגנון סבבי מיפוי דינמיים: היועץ מגדיר 3 פרמטרים לתקופה ופותח סבב, מורים נכנסים לפורטל נפרד (`/portal`) עם תעודת זהות בלבד וממלאים טבלה ממורכזת לכל כיתה/מקצוע שלהם עם שמירה אוטומטית, והיועץ עוקב אחרי אחוזי השלמה בזמן אמת. ראו פירוט מלא למטה.
- **פגישות (עמוד הבית)** - תיעוד וסיכום פגישות עם תלמידים: תאריך, שיוך למספר תלמידים, משתתפים, כותרת, סיכום בעריכת טקסט עשיר (הדגשה, כותרות, רשימות), וקבצים מצורפים (גרירה או בחירה). טבלת כל הפגישות עם עריכה.
- **לוח תלמידים** - שם, תעודת זהות (עם אימות תקינות), שיבוץ לכיתה, הערות, ופרופיל קשר מלא (טלפונים, דוא"ל, כתובת, פרטי שני אפוטרופוסים) - ניתן להזין ידנית או לייבא מאקסל. חיפוש וסינון לפי כיתה. תיק אישי לכל תלמיד הכולל גם פניות וגם פגישות שקשורות אליו.
- **מורים** - ניהול צוות הוראה עם שיוך חכם לכיתות ומקצועות (מורה אחד יכול ללמד כמה מקצועות בכמה כיתות). תיק אישי לכל מורה עם פרטי קשר מלאים ורשימת השיוכים. ניתן לייבא מאקסל.
- **לוח כיתות** - הגדרת רשימת הכיתות, כל כיתה עם צבע מזהה שנבחר מפלטה קבועה.
- **לוח פניות** - לוח קנבן עם שלוש עמודות (חדש / בטיפול / טופל), כל פנייה מקושרת לתלמיד ספציפי מהרשימה, עם נושא, תיאור ורמת עדיפות. גרירה בין עמודות מעדכנת את הסטטוס.
- **ייבוא מאקסל** - ייבוא תלמידים, כיתות ומורים מקבצי אקסל (ראו פירוט למטה).
- **התחברות** - משתמש יחיד עם סיסמה (ללא מערכת הרשאות מורכבת).

הנתונים (כולל קבצים מצורפים לפגישות) נשמרים באותה תיקיית נתונים מקומית - SQLite ל-`data/crm.db`, קבצים ל-`data/uploads/meetings/` - ללא צורך בשירות חיצוני.

## מיפוי תלמידים - פורטל מורים + לוח בקרה ליועץ

### איך זה עובד

1. **היועץ** נכנס ל-`/mapping` (בתפריט "מיפוי"), מקליד 3 שמות פרמטרים לתקופה הנוכחית, ולוחץ "פתח סבב מיפוי חדש". זה משבית את הסבב הקודם (אם היה) ופותח סבב חדש - תשובות ישנות נשארות בהיסטוריה, לא נמחקות.
2. **מורה** נכנס ל-`/portal` (כתובת נפרדת, לא דרך תפריט היועץ - יש לשתף אותה עם המורים ישירות), מזין רק תעודת זהות (**בלי סיסמה**), רואה כרטיסיות של הכיתות/מקצועות המשויכים אליו בלבד, ובכל כרטיס - טבלה עם כל תלמידי הכיתה. לחיצה על אחד מ-4 הכפתורים הצבעוניים (מעל הממוצע / בממוצע / מתחת לממוצע / קושי משמעותי) שומרת מיד, בלי כפתור "שמור" - עם וי ירוק ליד שם התלמיד לאישור.
3. **היועץ** חוזר ל-`/mapping` ורואה טבלת מעקב: אחוז השלמה לכל שילוב כיתה־מקצוע־מורה.

### נקודות חשובות לגבי אבטחה

- **הכניסה לפורטל המורים היא זיהוי, לא אימות מאובטח.** תעודת זהות אינה סוד - מורה שיודע (או מנחש) ת.ז של עמית יכול תיאורטית להיכנס בתור אותו עמית. זו פשרה מכוונת לטובת כניסה מהירה בלי סיסמה, כפי שהתבקש, ומתאימה לכלי פנימי בסיכון נמוך. אם בעתיד תרצו רמת אבטחה גבוהה יותר, אפשר להוסיף קוד חד-פעמי שנשלח למייל המורה, אבל זה משנה את חוויית "כניסה מהירה".
- מה שכן מוגן: כל בקשת שמירה בשרת מוודאת שהשיוך (כיתה+מקצוע) באמת שייך למורה המחובר, וכל תלמיד באמת שייך לאותה כיתה - מורה לא יכול לשמור נתונים לכיתה/מקצוע שהוא לא משויך אליהם גם אם הוא מנסה ישירות מול ה-API.
- מסך `/portal` וכל ה-API שלו (`/api/portal/*`) **אינם דורשים את סיסמת היועץ** - הם מוגנים בנפרד דרך `proxy.js`. שאר המערכת (`/mapping`, `/teachers` וכו') עדיין דורשת התחברות עם סיסמת היועץ הרגילה.

### כדי שמורה יוכל להיכנס לפורטל

חובה שיהיה לו תעודת זהות רשומה בעמוד עריכת המורה (`/teachers`) - בלי זה הוא לא יימצא בזיהוי. ניתן להזין ידנית, או לייבא ת.ז דרך ייבוא מורים מאקסל (עמודה "ת.ז" / 'ת"ז' / "תעודת זהות" מזוהה אוטומטית אם קיימת בקובץ).



בעמוד **ייבוא** (או מכפתור "ייבוא מאקסל" בעמוד התלמידים) ניתן להעלות קובץ אלפון תלמידים (.xls/.xlsx) כפי שמיוצא ממערכות בתי ספר. המערכת:

1. קוראת את כל עמודות התלמיד: שם, ת.ז, תאריך לידה, כתובת, טלפונים, דוא"ל, ופרטי שני האפוטרופוסים (שם, ת.ז, טלפון, כתובת, דוא"ל, סוג קרבה).
2. מנסה לשייך את עמודת "כיתת אם" (למשל "א' - 1") **לכיתה קיימת בלבד** לפי שם מדויק. כיתה שלא קיימת במערכת לא נוצרת אוטומטית - התלמיד מיובא בלי שיוך לכיתה, עם אזהרה בתצוגה המקדימה.
3. מציגה תצוגה מקדימה מלאה לפני השמירה בפועל, עם סימון שורות עם שגיאה (שם/ת.ז חסרים, ת.ז כפולה בקובץ) שלא ייובאו, ואזהרות (כיתה לא מוכרת, ת.ז שלא עברה בדיקת ביקורת) שכן ייובאו אך מסומנות לתשומת לב.
4. תלמיד עם ת.ז שכבר קיימת במערכת **מתעדכן** (לא משוכפל) - כך שניתן לייבא את אותו קובץ שוב ושוב כדי לסנכרן שינויים.

חשוב: **אין במערכת ייבוא מורים** כרגע - קובץ האלפון הנתמך מכיל רק נתוני תלמידים. אם יתווסף בעתיד קובץ נפרד לאלפון עובדי הוראה, ניתן להרחיב את `lib/importStudents.js` בהתאם.

### ייבוא מורים

בטאב "ייבוא מורים" באותו עמוד. **בשונה מייבוא התלמידים, זה לא נבדק מול קובץ אמיתי** - מיפוי העמודות (שם מורה / מייל / טלפון / כתובת / כיתה / מקצוע, בכמה ניסוחים נפוצים) הוא בגדר ניחוש סביר. אם קובץ המורים האמיתי שלכם לא מזוהה נכון, שתפו אותו כדי שנתאים את `lib/importTeachers.js` בדיוק אליו (כפי שנעשה עבור התלמידים).

מכיוון שלמורה יכולים להיות כמה שילובי כיתה/מקצוע, קובץ שבו אותו מורה מופיע בכמה שורות (פעם לכל שילוב) יאוחד אוטומטית לתיק מורה אחד עם כל השיוכים - האיחוד מתבצע לפי דוא"ל אם קיים, אחרת לפי שם מלא. מקצועות חדשים נוצרים אוטומטית לפי הצורך, ולא נוצרים כפילויות אם המקצוע כבר קיים.

### הערת אבטחה - ספריית פענוח האקסל

הייבוא משתמש בספריית `xlsx` (SheetJS) מ-npm, שיש לה אזהרת אבטחה ידועה (Prototype Pollution / ReDoS - ללא תיקון זמין ב-npm כרגע). מכיוון שזהו כלי פנימי למשתמש יחיד וקבצי הייבוא מגיעים רק מכם עצמכם (לא מהעלאות ציבוריות), הסיכון בפועל נמוך. אם תרצו להחמיר אבטחה, ניתן להחליף בגרסה המתוקנת המופצת ישירות מ-SheetJS:

```bash
npm remove xlsx
npm install https://cdn.sheetjs.com/xlsx-0.20.3/xlsx-0.20.3.tgz
```
(דורש גישה לאינטרנט בזמן ההתקנה; לא נבדק בסביבת הפיתוח הנוכחית עקב הגבלות רשת).


## הרצה מקומית

דורש **Node.js גרסה 22.5 ומעלה**.

```bash
npm install
```

צרו קובץ `.env.local` (ראו `.env.example`):

```bash
cp .env.example .env.local
```

צרו hash לסיסמת הכניסה שלכם:

```bash
npm run hash-password -- "הסיסמה-שלי"
```

העתיקו את הפלט לתוך `ADMIN_PASSWORD_HASH` ב-`.env.local`, ומלאו גם `SESSION_SECRET` (מחרוזת אקראית ארוכה כלשהי).

הרצה:

```bash
npm run dev
```

האתר יעלה בכתובת http://localhost:3000 ויפנה אוטומטית למסך ההתחברות.

## פריסה ל-Render

1. **דחפו את הפרויקט ל-GitHub.**

2. **צרו Web Service חדש** ב-Render, וחברו אותו לריפו.

3. **הגדרות בנייה:**
   - Build Command: `npm install && npm run build`
   - Start Command: `npm start`

4. **הוסיפו Persistent Disk** (חשוב! בלי זה הנתונים יימחקו בכל דיפלוי):
   - בעמוד השירות ב-Render: Disks → Add Disk
   - Mount Path: `/var/data`
   - גודל: 1GB מספיק בהחלט להתחלה

5. **משתני סביבה** (Environment):
   - `DATA_DIR` = `/var/data`
   - `SESSION_SECRET` = מחרוזת אקראית ארוכה משלכם
   - `ADMIN_PASSWORD_HASH` = ה-hash שיצרתם בשלב הקודם (אפשר גם להריץ את הסקריפט מקומית ולהעתיק את הפלט)
   - `NODE_ENV` = `production`

6. **Deploy.** בפעם הראשונה שהמערכת עולה היא תיצור את קובץ מסד הנתונים בתוך `/var/data` ואת כל הטבלאות הנדרשות אוטומטית.

### שינוי סיסמה בעתיד

הריצו מקומית `npm run hash-password -- "סיסמה-חדשה"`, והחליפו את הערך של `ADMIN_PASSWORD_HASH` בהגדרות הסביבה ב-Render.

## הערות טכניות

- מסד הנתונים נפתח רק ב-runtime (לא בזמן `next build`) בעזרת Proxy עצלן ב-`lib/db.js`, כדי למנוע נעילת קובץ בזמן הבנייה.
- טבלת התלמידים עברה הרחבה (שדות פרופיל מלאים) לאחר הגרסה הראשונה. `lib/db.js` בודק בכל עלייה אילו עמודות חסרות ומוסיף אותן אוטומטית (`ALTER TABLE ADD COLUMN`) - אין צורך בהרצת מיגרציה ידנית, גם על מסד נתונים קיים ב-Render.
- ניהול ה-session מבוסס על Web Crypto (`crypto.subtle`) ולא על `node:crypto`, כדי שהקוד יעבוד גם ב-middleware שרץ ב-Edge runtime.
- מחיקת תלמיד מוחקת גם את כל הפניות והפגישות המקושרות אליו. מחיקת כיתה משאירה את התלמידים במערכת, רק מנתקת אותם מהכיתה.
- **קבצים מצורפים לפגישות נשמרים בדיסק** בתוך `DATA_DIR/uploads/meetings/`, לא במסד הנתונים - כך שהם דורשים את אותו Persistent Disk כמו מסד הנתונים. אם מתעדים הרבה פגישות עם קבצים כבדים (סריקות, תמונות), שווה להגדיל את גודל ה-Disk ב-Render מעבר ל-1GB המומלץ בהתחלה. גודל מקסימלי לקובץ בודד: 20MB (ניתן לשנות ב-`app/api/meetings/[id]/attachments/route.js`).
